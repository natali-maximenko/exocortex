# С чего начать разработчику IWE

> **Это единственная ссылка, которая нужна новичку.** Прочитай сверху вниз - за 10 минут поймёшь, что делать, и сделаешь первую задачу.

Разработка в IWE идёт через **конвейер из 6 станций** (Постановка → Открытие → Проектирование → Работа → Проверка → Закрытие — подробно в разделе [«Цикл одной задачи»](#цикл-одной-задачи-6-станций) ниже и в [гайде](developer-guide.md)). Каждая задача проходит путь от постановки до закрытия, и на выходе остаётся не только код, но и зафиксированное знание (это называется **двойной выход**). Ты - разработчик уровня T4+, берёшь карточку задачи, проходишь станции, отправляешь Pull Request на проверку.

## Кто ты в этой схеме

Есть два способа помогать платформе, не путай их:

- **Просто оставить замечание** (это делают пользователи T1-T3) - прислал баг или идею, и кто-то другой решает, что с этим делать. Код ты не трогаешь.
- **Сделать работу** (это ты, T4+) - сам берёшь карточку, сам пишешь код и знание, сам отправляешь Pull Request. Твоя работа вливается в платформу.

Короче: одни **говорят, что починить**, ты **чинишь**. Это и есть разница между волонтёром с замечанием и разработчиком команды.

## Твой первый день - 3 шага

1. **Пойми правила игры.** Прочитай [гайд разработчика](developer-guide.md) - там 6 станций на одной странице, что такое двойной выход и кто принимает твой код. 5 минут.
2. **Возьми шаблон задачи.** Скопируй [шаблон карточки](card-template.md) в свою папку задач и заполни: что делаешь, что появится в коде, какое знание зафиксируешь.
3. **Пройди 6 станций и отправь Pull Request.** Карточка ведёт тебя по станциям. На выходе - код + capture (различение или заметка в памяти). Merge принимает ведущий разработчик.

## Делай это в своём IWE

Конвейер не нужно проходить вручную. Ты делаешь его **внутри своего IWE** - твоего рабочего пространства с агентом (Claude или Kimi), памятью и протоколами. Поставь его один раз, и дальше агент ведёт тебя по станциям сам.

1. **Поставь свой IWE.** Запусти установку шаблона (`bash setup.sh`). У тебя появятся: агент, инструкции, этот гайд и шаблон карточки - локально, под твоим аккаунтом. Все разработчики команды ставят один шаблон, поэтому говорят на одном языке.
2. **Загрузи рабочие хранилища команды.** Склонируй в своё пространство (`~/IWE/`) те репозитории, где будешь делать задачи: код сервиса и базу знаний (Pack). Без них агенту негде работать. Перед каждой задачей он сам подтянет свежую версию (`git pull`), чтобы ты не правил устаревший код. Работаешь в общем репозитории с другими - агент отметит твои активные файлы, чтобы не было конфликтов. **Если ты улучшаешь сам шаблон** (а не продукт команды) - целевой репозиторий = твой форк `FMT-exocortex-template`, отдельный «сервис» клонировать не нужно.
3. **Открывай задачу через агента (WP Gate).** Скажи агенту, что делаешь. Он сам проведёт ритуал открытия: объявит роль, работу, класс верификации, оценку и модель, и дождётся твоего согласия. Это станция «Открытие».
4. **Агент ведёт по станциям.** Каркас он знает из своих инструкций (ОРЗ - Открытие/Работа/Закрытие, WP Gate, двойной выход). Полную схему 6 станций он берёт из [гайда](developer-guide.md) - в начале работы попроси его прочитать гайд, тогда он не даст пропустить проверку и не забудет про capture. Код пишешь ты, агент помогает и проверяет.
5. **Знание копится в твоей памяти.** Каждое различение, урок и заметка (capture) ложится в твой экзокортекс (внешнюю память IWE). Ничего не теряется между задачами - в следующий раз агент это вспомнит.
6. **Перед Pull Request - проверка руками агента.** Он прогонит чеклист или тесты по классу верификации. Зелено - отправляешь Pull Request.

> **Главное:** твой IWE - не редактор кода, а рабочее место, которое помнит твою работу и держит дисциплину конвейера за тебя. Чем больше задач проходишь, тем умнее становится твоя личная память.

## Цикл одной задачи (6 станций)

**Постановка → Открытие → Проектирование → Работа → Проверка → Закрытие.**

Подробно каждая станция - в [гайде](developer-guide.md). Коротко: оформил задачу → согласовал с пилотом (WP Gate) → спроектировал → сделал код **вместе** с записью знания → проверил по классу верификации → Pull Request и merge.

**Двойной выход - это закон.** Задача, после которой остался только код без зафиксированного знания, считается **незакрытой**.

Когда откроешь Pull Request, подставится шаблон с чеклистом 6 станций - заполни его честно. По нему ревьюер сразу видит, что конвейер пройден.

## Барьер входа и кто принимает код

- Конвейер - для разработчиков **T4+**. Если твой уровень ниже, путь наверх описан в [LEARNING-PATH.md](../LEARNING-PATH.md) (это программы развития, не этот конвейер).
- **Merge принимает только** ведущий разработчик (TD1+TA4) или пилот. Никто другой - без явного делегирования.
- Застрял дольше оценки? Не буксуй тихо - эскалируй ведущему разработчику или пилоту.
